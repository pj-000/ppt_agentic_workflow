# 风险控制与最小可发表版本

## 1. 最大风险

### 风险 1：被认为只是工程系统

如果论文写成：

```text
我们做了一个可以交互修改 PPT 的系统。
```

风险很高。

解决：

```text
必须强调 Revision Index、定位算法、编辑规划和 benchmark。
```

---

### 风险 2：被认为只是 Prompt Engineering

如果方法是：

```text
用户说一句话 → LLM 判断改哪页 → LLM 修改 PPT
```

不够 CCF-B。

解决：

```text
1. 构建结构化 Index；
2. 用多信号 scoring 做定位；
3. 定义编辑操作空间；
4. 做消融实验证明各层 Index 有用。
```

---

### 风险 3：完整 PPT 后端不适合开源

解决：

```text
1. 论文核心不依赖私有后端；
2. 提供 lightweight executor；
3. 开源 index、benchmark、evaluation；
4. 内部后端只作为 optional adapter 或高质量 demo。
```

---

### 风险 4：用户研究不充分

IUI 很看重 human-centric evidence。

解决：

```text
最低做 8–12 人；
理想做 12–18 人；
任务设计真实；
有对照条件；
记录定量和定性反馈。
```

---

### 风险 5：数据集规模太小

解决：

```text
最低 300 条单轮任务；
理想 500 条单轮任务；
至少 30 组多轮任务；
每类任务至少 20–30 条。
```

---

## 2. 最小可发表版本（MVP Paper）

如果时间不够，至少完成：

```text
1. Revision Index schema；
2. PPT parser；
3. Intent parser；
4. Slide/element localization；
5. Edit planner；
6. Lightweight executor 支持 5–7 种操作；
7. 300 条 benchmark；
8. 3 个 baseline；
9. 3 个消融；
10. 8–12 人用户研究；
11. 2 个 case study。
```

MVP 操作集合：

```text
REWRITE_TEXT_BLOCK
ADD_SLIDE
DELETE_SLIDE
SPLIT_SLIDE
ADD_TEXT_BLOCK
MERGE_SLIDES
REORDER_SLIDE
```

MVP baseline：

```text
Keyword Search
Embedding Retrieval
LLM Direct Locate/Plan
Full Regeneration 或 LLM Direct Revision
```

---

## 3. 理想版本

```text
1. 40–60 套 PPT；
2. 500 条单轮任务；
3. 50–80 组多轮任务；
4. 5 个 baseline；
5. 5 个消融；
6. 12–18 人用户研究；
7. 可运行 UI demo；
8. 匿名开源 subset；
9. demo video；
10. 完整论文图表。
```

---

## 4. 投稿前必须有的证据

### Computational evidence

```text
- localization accuracy；
- operation prediction；
- ablation；
- multi-turn consistency；
- revision quality。
```

### Human-centric evidence

```text
- perceived control；
- trust；
- satisfaction；
- completion time；
- user preference；
- qualitative feedback。
```

---

## 5. 备选投稿策略

### 主路线：IUI 2027

适合版本：

```text
有 UI demo + 用户研究 + benchmark。
```

### 备选路线：CIKM

如果用户研究没做出来，但 index 和 benchmark 做得扎实，可改投 CIKM 风格：

```text
强调 indexing、retrieval、benchmark、localization。
```

### 备选路线：ICME / ICMR

如果视觉/布局评估做得更强，可转多媒体方向：

```text
强调 slide visual understanding、layout revision、multimodal editing。
```

---

## 6. 对老师的风险说明

可以这样说：

> 这个方向不训练模型，但 IUI 并不要求必须训练模型。风险主要在于如果只做系统 demo 会不够。因此我会把核心贡献放在 Revision Index、页面/元素定位、编辑规划和 benchmark 上，同时补用户研究。即使完整 PPT 后端不适合开源，我也会提供 lightweight executor 和匿名化评测集，保证论文核心可复现。

---

## 7. 最终底线

如果到 7 月底还没有完成：

```text
- benchmark；
- baseline；
- ablation；
- 用户研究；
```

就不要硬投 IUI 2027 full paper。

可以转为：

```text
1. 先写技术报告或 arXiv；
2. 扩展到 CIKM/IUI 下一轮；
3. 或先投 workshop/demo，但不要当 CCF-B 正式论文。
```

---

## 8. 成功标准

达到 CCF-B 竞争力的最低成功标准：

```text
1. 老师能清楚理解创新不是 PPT 生成，而是 Revision Index。
2. Codex 能按 Phase 完成可运行原型。
3. benchmark 能量化证明 index 比 keyword/embedding/LLM direct 更好。
4. 用户研究能证明系统比重生成更可控、更省时或更少误改。
5. 论文中能讲清楚 computational + human-centric 双重贡献。
```

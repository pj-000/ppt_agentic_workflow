# 仓库改造与系统架构

## 1. 当前仓库基础

目标仓库：

```text
https://github.com/pj-000/directionai-agent-backend
```

当前公开结构大致为：

```text
ppt_backend/
  独立 CLI 和 FastAPI 服务入口

runtime/
  原 PPT 生成 runtime、prompt 模板、vendor skill 和工作区

outputs/
  建议的独立输出目录

README.md
pyproject.toml
.env.example
```

当前项目能力包括：

```text
- CLI 直接生成 PPT
- 带文档生成
- 两阶段 outline/from-outline 生成
- FastAPI 服务
- 上传文档
- 生成 PPT
- 流式大纲
- 流式 from-outline 生成
- PPT 评估
- PPT 下载和预览
```

新方向不需要推翻这些能力。

---

## 2. 改造原则

### 原则 1：不破坏原有后端

不要直接重写 `ppt_backend/` 和 `runtime/`。

原有能力应保持可运行：

```text
uv run ppt-backend generate ...
uv run ppt-backend outline ...
uv run ppt-backend from-outline ...
uv run ppt-backend serve ...
```

### 原则 2：新增独立研究模块

新增目录：

```text
slide_revise_agent/
```

作为论文核心模块。

### 原则 3：现有 PPT 流程作为 backend adapter

原来的完整 PPT 生成流程不适合开源，所以论文中不要依赖它作为核心。它只作为：

```text
InternalBackendAdapter
```

同时实现：

```text
LightweightExecutor
```

用于复现实验。

### 原则 4：核心可开源部分

建议开源：

```text
- Revision Index schema
- PPT parser
- index builder
- localization
- edit planner
- lightweight executor
- benchmark loader
- evaluation scripts
```

不必开源：

```text
- 私有 prompt
- 私有 skill adapter
- 内部完整 PPT 生成 pipeline
- API key / 模型配置
- 业务后端细节
```

## 4. 系统总流程

```text
User Interaction
      ↓
Intent Parsing
      ↓
Revision Index Query
      ↓
Slide / Element Localization
      ↓
Edit Planning
      ↓
Local Revision Execution
      ↓
Result Verification
      ↓
Index Update
      ↓
Revised PPT + Explanation
```

---

## 5. 输入与输出

### 输入

```text
1. existing PPTX
2. user revision request
3. interaction context
   - selected_slide_id
   - selected_element_ids
   - current_view
   - previous turns
4. optional outline
5. optional source document
```

### 输出

```text
1. affected slides
2. affected elements
3. edit plan
4. revised PPTX
5. updated Revision Index
6. explanation
7. evaluation report
```

## 6. 模块职责

## 6.1 Parser

负责解析 PPT：

```text
- 每页标题
- 每页正文文本
- 每页元素
- 元素类型
- 元素位置 bbox
- 字号、颜色、样式
- 图片、shape、表格、图表
- 截图 preview
```

输出：

```text
DeckIndex 初始版本
```

---

## 6.2 Index Builder

负责构建：

```text
- Slide Semantic Index
- Element Index
- Dependency Index
- Operation Index
```

---

## 6.3 Interaction Manager

负责处理用户交互上下文：

```text
- 当前选中页面
- 当前选中元素
- 历史修改记录
- 指代解析：这页、这里、刚才那页、前面那部分
```

---

## 6.4 Intent Parser

将自然语言请求解析为：

```text
- intent_type
- target_scope
- explicit_references
- content_requirements
- style_requirements
- candidate_operation_templates
```

---

## 6.5 Locator

根据 Revision Index 输出：

```text
- affected_slides
- affected_elements
- confidence
- reason
```

---

## 6.6 Edit Planner

根据定位结果生成 edit plan：

```text
REWRITE_TEXT_BLOCK
ADD_TEXT_BLOCK
DELETE_TEXT_BLOCK
ADD_SLIDE
DELETE_SLIDE
SPLIT_SLIDE
MERGE_SLIDES
REORDER_SLIDE
CHANGE_STYLE
UPDATE_SUMMARY
```

---

## 6.7 Executor

执行局部修改。

分两类：

```text
Lightweight executor：可复现，用 python-pptx 或 PptxGenJS 实现基础操作。
Internal backend adapter：调用当前项目现有后端，生成更高质量 demo。
```

---

## 6.8 Verification

验证修改结果：

```text
- 用户意图是否完成
- 未受影响页面是否保持不变
- 风格是否一致
- PPT 是否可打开
- index 是否更新正确
```

---

## 7. 与原有 FastAPI 的关系

建议新增一组 API，不影响原有接口：

```text
POST /slide-revise/index/build
POST /slide-revise/intent/parse
POST /slide-revise/localize
POST /slide-revise/plan
POST /slide-revise/execute
POST /slide-revise/revise
GET  /slide-revise/index/{deck_id}
GET  /slide-revise/report/{run_id}
```

也可以先不接入 FastAPI，先做 CLI：

```bash
uv run python -m slide_revise_agent.cli build-index --ppt input.pptx --out deck_index.json
uv run python -m slide_revise_agent.cli revise --ppt input.pptx --instruction "这页太满了，拆成两页" --selected-slide 5
```

## 10. 重要约束

Codex 实现时必须遵守：

```text
1. 不修改 runtime/ 内部核心生成逻辑，除非明确要求；
2. 不删除原有接口；
3. 新模块先独立运行；
4. 所有实验输出放到 experiments/slide_revise/outputs；
5. 所有新 schema 使用 Pydantic；
6. 所有函数都要有类型标注；
7. 每个 Phase 都要写最小测试；
8. 每次修改后保存 run log。
```

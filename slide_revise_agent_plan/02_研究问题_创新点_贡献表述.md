# 研究问题、创新点与贡献表述

## 1. 研究背景

现有 PPT 生成系统主要关注一次性生成整套 PPT：

```text
用户输入主题/文档 → 系统生成完整 PPT
```

但真实使用场景中，PPT 创作通常是反复修改的：

```text
已有 PPT → 用户发现问题 → 局部修改 → 再检查 → 再修改
```

常见用户需求包括：

```text
- 这页太满了，拆成两页。
- 这里加一个例子。
- 把这几页改得更适合本科生。
- 把刚才新增的那页讲得简单一点。
- 删除高级推导部分。
- 先讲例子，再讲公式。
- 整体改成答辩风格。
- 把这个流程图改清楚一点。
```

如果直接重新生成整套 PPT，容易导致：

```text
1. 无关页面被误改；
2. 原有风格被破坏；
3. 修改范围不可控；
4. 多轮交互后上下文丢失；
5. 用户不知道系统为什么改这些页。
```

---

## 2. 核心研究问题

最终研究问题：

> **在交互式 PPT 修改场景中，如何根据用户的修改意图，准确定位需要修改的页面、元素和内容，并规划最小化、可控、可解释的局部编辑操作？**

任务定义应保持为：给定已有 PPT、用户修改请求、当前交互上下文和历史修改记录，系统需要定位受影响页面和元素，生成编辑操作计划，并执行最小化局部修改，同时保持未受影响页面、原有视觉风格和整套 PPT 的逻辑连贯性。

拆成四个子问题：

### RQ1：定位问题

用户说“这页”“这里”“刚才新增的那页”“Transformer 那部分”时，系统如何知道指的是哪一页、哪个元素、哪个知识点？

### RQ2：修改范围问题

系统如何判断这次修改应该影响：

```text
- 一个文本框？
- 一整页？
- 多页？
- 新增一页？
- 目录页和总结页也要同步更新？
```

### RQ3：编辑操作问题

定位后，系统如何判断应该采取什么操作：

```text
- 重写文本
- 扩写概念
- 压缩内容
- 新增例题
- 拆页
- 合并页
- 调整顺序
- 修改风格
```

### RQ4：多轮一致性问题

修改完成后，系统如何更新索引，使下一轮用户说“刚才那页”“这个例子”“前面那部分”时仍能正确定位？

---

## 3. 任务定义

### 输入

```text
D: existing slide deck
q: user revision request
c: interaction context
h: interaction history
I: current Revision Index
```

其中：

```text
D 包含 slide text、layout、images、elements 等信息；
q 是自然语言修改请求；
c 包含 selected_slide_id、selected_element_id、current_view 等；
h 是历史修改记录；
I 是当前 deck 的多粒度 Revision Index。
```

### 输出

```text
T_s: affected slides
T_e: affected elements
O: edit operation sequence
D': revised deck
I': updated Revision Index
E: explanation
```

### 目标

```text
1. 满足用户修改意图；
2. 尽量只修改必要页面和元素；
3. 保持未受影响页面不变；
4. 保持整体风格与逻辑一致；
5. 支持多轮交互。
```

---

## 4. 核心创新点

## 创新点 1：交互式 PPT 局部修改任务

现有工作多关注一键生成整套 PPT。本工作关注真实创作中的 iterative local revision。

贡献表述：

> We formulate interactive slide deck revision as a task that maps user revision requests and interaction contexts to affected slides, affected elements, and minimal edit operations.

中文表述：

> 本工作提出交互式 PPT 局部修改任务，将用户修改请求和交互上下文映射为受影响页面、受影响元素和最小化编辑操作。

---

## 创新点 2：Multi-granularity Revision Index

提出多粒度 Revision Index，包括：

```text
Interaction Context Index
Slide Semantic Index
Element Index
Dependency Index
Operation Index
```

它不仅支持“找第几页”，还支持：

```text
- 找具体元素；
- 理解交互指代；
- 判断页面依赖；
- 规划编辑操作；
- 多轮更新。
```

贡献表述：

> We introduce a Multi-granularity Revision Index that jointly represents interaction contexts, slide semantics, visual/textual elements, inter-slide dependencies, and edit operation affordances.

---

## 创新点 3：Index-guided Localization

我们将交互式修改中的页面/元素定位形式化为一个 training-free 的多信号排序问题，并提出基于多信号融合的页面/元素定位方法：

```text
TargetScore(s) =
  α · ExplicitReferenceScore(s)
+ β · SelectionContextScore(s)
+ γ · SemanticSimilarityScore(s)
+ δ · ElementRelevanceScore(s)
+ ε · DependencyScore(s)
+ ζ · InteractionHistoryScore(s)
```

这个方法不需要训练模型，可以用规则权重或验证集调参。

贡献表述：

> We develop an index-guided localization method that combines explicit references, selection context, semantic similarity, element relevance, dependency signals, and interaction history to identify affected slides and elements.

---

## 创新点 4：Index-guided Edit Planning

定位页面/元素后，系统根据用户意图、页面密度、元素结构、依赖关系和操作索引生成编辑计划。

示例：

```text
用户说：这页太满了
→ 如果 word_count 高、元素密度高，选择 SPLIT_SLIDE；
→ 如果只有少量冗余 bullet，选择 REWRITE_TEXT_BLOCK，并在参数中标明 content_intent="compress_content"。

用户说：这里加个例子
→ 如果当前页有空间，选择 ADD_TEXT_BLOCK；
→ 如果当前页已经很满，选择 ADD_SLIDE after current slide。
```

贡献表述：

> We propose an index-guided edit planner that selects local edit operations under preservation constraints, enabling controlled revisions without regenerating the full deck.

---

## 创新点 5：Dynamic Index Update for Multi-turn Revision

每次修改后，系统重新解析受影响页面并更新 Index：

```text
SlideIndex
ElementIndex
DependencyIndex
InteractionHistory
```

使多轮交互可持续。

贡献表述：

> We maintain Revision Index dynamically after local edits, enabling multi-turn references such as “the slide just added” and reducing error accumulation.

---

## 5. 最终论文贡献写法

论文中建议写四条 contribution：

```text
1. We introduce the task of interactive local slide deck revision, where systems must localize affected slides/elements and plan minimal edits from user interactions.

2. We propose SlideReviseAgent, a training-free interactive revision framework built around a Multi-granularity Revision Index over interaction contexts, slide semantics, elements, dependencies, and operations.

3. We design index-guided localization, edit planning, and dynamic index update algorithms to support controllable and multi-turn slide revision.

4. We construct SlideReviseBench-ZH and evaluate our method through offline localization/planning experiments, ablation studies, revision quality evaluation, and a user study.
```

中文汇报版：

```text
1. 提出交互式 PPT 局部修改任务；
2. 提出多粒度 Revision Index；
3. 提出基于 Index 的页面/元素定位、编辑规划和动态更新方法；
4. 构建 SlideReviseBench-ZH，并通过离线实验、消融实验和用户研究验证方法有效性。
```

## 6. 这个工作是否需要训练模型？

不需要。

但要强调：

```text
Training-free 不等于 method-free。
```

本工作的技术贡献不是训练新模型，而是：

```text
- 任务形式化；
- 多粒度 Revision Index 结构；
- 页面/元素定位算法；
- 编辑操作规划算法；
- 多轮动态更新机制；
- benchmark 和用户研究。
```

不要说：

```text
我们只是调用大模型修改 PPT。
```

要说：

```text
我们把大模型作为意图理解和文本生成工具，把 PPT 后端作为执行器，论文核心是 Revision Index 和 index-guided revision workflow。
```

---

## 7. 不该作为创新点的内容

不要把下面内容写成主创新：

| 不建议主打 | 原因 |
|---|---|
| 会生成 PPT | 现有 PPT skill / PPT 生成工具已有 |
| 调用 Anthropic PPT Skill | 容易被认为是封装现有能力 |
| Prompt 优化 | 不够 CCF-B |
| 简单系统 Demo | 缺少方法和实验 |
| 单纯大纲修改 | 交互入口太窄 |
| 整套 PPT 自动生成 | 不适合当前开源约束，也不是老师建议重点 |

---

## 8. 最终一句话

> **SlideReviseAgent 是一个不训练新模型的交互式 PPT 局部修改 Agent。它的核心贡献是多粒度 Revision Index，以及基于 Index 的页面/元素定位、编辑操作规划和动态更新机制，用于支持用户可控、低干扰、多轮一致的 PPT 修改。**

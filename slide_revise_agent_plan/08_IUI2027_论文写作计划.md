## 4. Section 1：Introduction

### 要讲的逻辑

```text
1. Slide authoring is iterative, not one-shot.
2. Existing slide generation systems focus on deck creation.
3. In real use, users need local revision: add examples, split dense slides, simplify language, reorder sections.
4. Full regeneration damages existing slides and reduces user control.
5. Direct LLM revision often lacks precise localization and edit planning.
6. We propose SlideReviseAgent with a Multi-granularity Revision Index.
```

### Introduction 贡献句

```text
We make four contributions:

1. We formulate interactive local slide deck revision as a task of mapping user revision requests and interaction contexts to affected slides/elements and minimal edit operations.

2. We propose SlideReviseAgent, a training-free interactive revision framework built around a Multi-granularity Revision Index.

3. We design index-guided localization, edit planning, and dynamic index update methods for controllable multi-turn revision.

4. We construct SlideReviseBench-ZH and evaluate SlideReviseAgent through offline experiments, ablation studies, revision quality evaluation, and a user study.
```

## 6. Section 3：Formative Motivation and Design Goals

如果时间允许，做 5–8 人简短访谈：

```text
访谈对象：经常制作 PPT 的研究生/助教/老师。
问题：
1. 你修改 PPT 时最常见的需求是什么？
2. 你是否愿意让 AI 重新生成整套 PPT？
3. 你最担心 AI 改 PPT 的什么问题？
4. 你希望 AI 如何解释它修改了哪些页？
```

得到设计目标：

```text
DG1: Locality — only revise necessary slides/elements.
DG2: Controllability — users can steer revision through natural language and selection.
DG3: Explainability — system explains affected targets and operations.
DG4: Multi-turn consistency — system maintains references across turns.
DG5: Backend agnosticism — framework does not depend on one proprietary PPT backend.
```

如果来不及做访谈，可以写成 observed use cases + design goals，但 IUI 更推荐有 formative evidence。

## 7. Section 4：Task Definition

形式化：

```text
Given:
D = existing slide deck
q = user revision request
c = interaction context
h = interaction history
I = Revision Index

Predict:
T_s = affected slides
T_e = affected elements
O = edit operations

Generate:
D' = revised deck
I' = updated index
```

目标函数可以描述为：

```text
maximize task satisfaction
maximize preservation of unaffected slides
maximize coherence and style consistency
minimize unnecessary edits
```

中文定义：给定已有 PPT、用户修改请求、当前交互上下文和历史修改记录，系统需要定位受影响页面和元素，生成编辑操作计划，并执行最小化局部修改，同时保持未受影响页面、原有视觉风格和整套 PPT 的逻辑连贯性。


### 5.4 Index-guided Localization

介绍 scoring：

```text
TargetScore(s) =
  α · ExplicitReferenceScore(s)
+ β · SelectionContextScore(s)
+ γ · SemanticSimilarityScore(s)
+ δ · ElementRelevanceScore(s)
+ ε · DependencyScore(s)
+ ζ · InteractionHistoryScore(s)
```

我们将交互式修改中的页面/元素定位形式化为一个 training-free 的多信号排序问题。

### 5.5 Index-guided Edit Planning

介绍操作空间和规则/约束：

```text
SPLIT_SLIDE
ADD_SLIDE with slide_type="example" and content_intent="add_example"
REWRITE_TEXT_BLOCK
DELETE_TEXT_BLOCK
CHANGE_STYLE
```

## 9. Section 6：System Implementation

讲清楚：

```text
- PPT parser
- index builder
- locator
- edit planner
- lightweight executor
- optional backend adapter
- UI demo
```

前端界面不是论文的核心方法贡献，但为了 IUI 投稿、demo 展示和用户研究，需要实现一个轻量级 research prototype。界面支持上传 PPT、浏览页面缩略图、选择页面/元素、输入自然语言修改请求、展示系统定位结果、编辑计划和修改前后对比。

关键表述：

> Our framework is backend-agnostic. For reproducibility, we implement a lightweight executor that supports common local edit operations. The same index and planning modules can also call stronger slide editing backends through adapters.

避免说：

```text
我们依赖某个不可开源 PPT 生成流程。
```

## 14. Section 11：Limitations

诚实写：

```text
1. Lightweight executor cannot handle all complex designs.
2. Some visual/layout edits still require stronger backends.
3. Dataset is focused on Chinese educational/research PPTs.
4. User study scale is moderate.
5. LLM-based intent parsing may still fail on ambiguous instructions.
```

不要写成致命缺陷。强调：

> The proposed Revision Index and planning framework are backend-agnostic and can benefit from stronger executors.
